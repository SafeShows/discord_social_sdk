1//! Lobbies, guilds, and linking a lobby to a Discord channel.
2//!
3//! A lobby is a developer-controlled group of users that can chat and talk
4//! together. This module holds the [`Client`] half of the lobby API: creating
5//! and joining lobbies, looking up [`Lobby`] handles, observing lobby events,
6//! and wiring a lobby up to a real Discord channel.
7//!
8//! # Lobby secrets
9//!
10//! [`create_or_join_lobby`](Client::create_or_join_lobby) does not take a lobby
11//! ID — it takes a *secret*, a hard-to-guess string the game generates. Every
12//! user who joins with the same secret lands in the same lobby, and the first
13//! one to use it creates that lobby. That is the whole membership mechanism: no
14//! server-side component is required, and possession of the secret *is* the
15//! invitation.
16//!
17//! Because of that, a secret should be exchanged only through a channel that
18//! already restricts who can see it. Discord's activity-invite and rich-presence
19//! systems both carry a secret string visible only to accepted party members,
20//! and are the recommended transport.
21//!
22//! Secrets expire after roughly 30 days. Past that point the old lobby can no
23//! longer be joined and the same secret will create a fresh one instead, so this
24//! flow is not suitable for long-lived lobbies — use the [server APIs] for those.
25//!
26//! [server APIs]: https://docs.discord.com/developers/resources/lobby
27//!
28//! # Lifetime
29//!
30//! Client-created lobbies auto-delete once idle (no connected users) for a few
31//! minutes — 5 by default, configurable up to 7 days through the server API.
32//! Only lobbies that have a secret can be left with
33//! [`leave_lobby`](Client::leave_lobby); lobbies created through the server API
34//! must be manipulated through the server API.
35//!
36//! # Channel linking
37//!
38//! A lobby can be linked to a Discord text channel so messages sent in one show
39//! up in the other. The flow is: [`user_guilds`](Client::user_guilds) to list
40//! the user's servers, [`guild_channels`](Client::guild_channels) to list the
41//! channels in the chosen one, then
42//! [`link_channel_to_lobby`](Client::link_channel_to_lobby).
43//!
44//! Only members holding the `CanLinkLobby` flag — settable solely through the
45//! server API — may link a lobby, which lets a game restrict linking to clan or
46//! guild leaders. Unlinking needs only that same flag, not any Discord
47//! permission on the channel.
48//!
49//! Discord permissions are **not** synchronised into the game: every member of a
50//! lobby can read and write the linked channel regardless of whether they could
51//! see it on Discord. See
52//! [`GuildChannel::is_viewable_and_writeable_by_all_members`] before offering
53//! private channels for linking.
54//!
55//! [`GuildChannel::is_viewable_and_writeable_by_all_members`]:
56//!     crate::channel::GuildChannel::is_viewable_and_writeable_by_all_members

67/// Borrow a slice of key/value pairs as two parallel `Discord_String` arrays.
68///
69/// `Discord_Properties` is `{ size, keys, values }` — two arrays indexed in
70/// lockstep, not a map. The returned vectors own nothing but the `Discord_String`
71/// descriptors; the bytes they point at still belong to `pairs`.
72fn properties_arrays(
73    pairs: &[(&str, &str)],
74) -> (Vec<sys::Discord_String>, Vec<sys::Discord_String>) {
75    let keys = pairs.iter().map(|(k, _)| string::borrow(k)).collect();
76    let values = pairs.iter().map(|(_, v)| string::borrow(v)).collect();
77    (keys, values)
78}
79
80/// Assemble a `Discord_Properties` from arrays produced by [`properties_arrays`].
81///
82/// # Safety
83///
84/// The result borrows both vectors and every string they describe. It must not
85/// outlive them, and may only be passed to SDK functions that copy their input
86/// before returning — which is what `Discord_Client_CreateOrJoinLobbyWithMetadata`
87/// does. Never pass the result to `Discord_FreeProperties`: that function frees
88/// SDK-allocated buffers, and none of this memory is SDK-allocated.
89fn properties(
90    keys: &mut [sys::Discord_String],
91    values: &mut [sys::Discord_String],
92) -> sys::Discord_Properties {
93    sys::Discord_Properties {
94        size: keys.len().min(values.len()),
95        keys: keys.as_mut_ptr(),
96        values: values.as_mut_ptr(),
97    }
98}

100/// Trampoline for `Discord_Client_CreateOrJoinLobbyCallback`.
101unsafe extern "C" fn create_or_join_tramp<F>(
102    result: *mut sys::Discord_ClientResult,
103    lobby_id: u64,
104    userdata: *mut c_void,
105) where
106    F: FnOnce(Result<u64>) + 'static,
107{
108    // SAFETY: `userdata` is the boxed `F` installed alongside this trampoline.
109    unsafe { callback::dispatch_once::<F>(userdata, |f| f(to_result(result).map(|()| lobby_id))) }
110}

146    /// Join the lobby identified by `secret`, creating it if it does not exist.
147    ///
148    /// Everyone who joins with the same secret ends up in the same lobby, and
149    /// the first to use it creates that lobby; on success the callback receives
150    /// the lobby's ID.
151    ///
152    /// The secret should be hard to guess, since possession of it *is* the
153    /// invitation. Discord recommends distributing it through the activity
154    /// invite and rich presence systems, which carry a secret string only
155    /// accepted party members can see.
156    ///
157    /// Secrets expire after roughly 30 days, after which the old lobby can no
158    /// longer be joined and the same secret creates a new one — so this is not
159    /// the right tool for long-lived lobbies. The lobby also auto-deletes once
160    /// idle, by default after 5 minutes with no connected user.
161    ///
162    /// Completes through [`run_callbacks`](crate::run_callbacks).
163    pub fn create_or_join_lobby<F>(&mut self, secret: &str, callback: F)
164    where
165        F: FnOnce(Result<u64>) + 'static,
166    {
167        // SAFETY: the secret is copied by the SDK during the call, and the SDK
168        // owns the boxed closure until it invokes `free_fn`.
169        unsafe {
170            sys::Discord_Client_CreateOrJoinLobby(
171                self.as_raw_mut(),
172                string::borrow(secret),
173                Some(create_or_join_tramp::<F>),
174                callback::free_fn::<Option<F>>(),
175                callback::once_userdata(callback),
176            )
177        }
178    }

180    /// [`create_or_join_lobby`](Self::create_or_join_lobby) with metadata attached.
181    ///
182    /// `lobby_metadata` is stored on the lobby and `member_metadata` on the
183    /// joining member; both are flat string key/value pairs readable by every
184    /// member through [`Lobby::metadata`](crate::lobby::Lobby::metadata) and
185    /// [`LobbyMember::metadata`](crate::lobby::LobbyMember::metadata). A typical
186    /// use is publishing each player's in-game ID so the whole lobby can map
187    /// Discord IDs onto game IDs.
188    ///
189    /// Later calls **overwrite** the lobby's and the member's metadata rather
190    /// than merging into it.
191    pub fn create_or_join_lobby_with_metadata<F>(
192        &mut self,
193        secret: &str,
194        lobby_metadata: &[(&str, &str)],
195        member_metadata: &[(&str, &str)],
196        callback: F,
197    ) where
198        F: FnOnce(Result<u64>) + 'static,
199    {
200        let (mut lobby_keys, mut lobby_values) = properties_arrays(lobby_metadata);
201        let (mut member_keys, mut member_values) = properties_arrays(member_metadata);
202        let lobby_props = properties(&mut lobby_keys, &mut lobby_values);
203        let member_props = properties(&mut member_keys, &mut member_values);
204        // SAFETY: both `Discord_Properties` values borrow the four vectors above
205        // and the caller's strings. All of them are still in scope when the call
206        // returns, and the SDK copies the properties during the call — the C++
207        // wrapper's `ConvertedProperties` frees its own copies as soon as the
208        // same call returns, which is what proves the copy happens. They are
209        // deliberately not passed to `Discord_FreeProperties`, which is only for
210        // SDK-allocated buffers.
211        unsafe {
212            sys::Discord_Client_CreateOrJoinLobbyWithMetadata(
213                self.as_raw_mut(),
214                string::borrow(secret),
215                lobby_props,
216                member_props,
217                Some(create_or_join_tramp::<F>),
218                callback::free_fn::<Option<F>>(),
219                callback::once_userdata(callback),
220            )
221        }
222    }

248    /// The lobby with the given ID, if the SDK has it loaded.
249    pub fn lobby(&self, lobby_id: u64) -> Option<Lobby> {
250        let mut raw = MaybeUninit::<sys::Discord_LobbyHandle>::uninit();
251        // SAFETY: the out-parameter is only initialised when the call returns
252        // true, and ownership of the handle then transfers to us.
253        unsafe {
254            if sys::Discord_Client_GetLobbyHandle(self.raw_ptr(), lobby_id, raw.as_mut_ptr()) {
255                Some(Lobby::from_raw(raw.assume_init()))
256            } else {
257                None
258            }
259        }
260    }

297    /// Every guild (Discord server) the current user is a member of.
298    ///
299    /// The first step of the channel-linking flow: use it to let the user pick
300    /// which server to link to, then call
301    /// [`guild_channels`](Self::guild_channels) for that guild.
302    pub fn user_guilds<F>(&mut self, callback: F)
303    where
304        F: FnOnce(Result<Vec<GuildMinimal>>) + 'static,
305    {
306        unsafe extern "C" fn tramp<F>(
307            result: *mut sys::Discord_ClientResult,
308            guilds: sys::Discord_GuildMinimalSpan,
309            userdata: *mut c_void,
310        ) where
311            F: FnOnce(Result<Vec<GuildMinimal>>) + 'static,
312        {
313// SAFETY: the span is transferred to us — the C++ wrapper adopts every
314            // element as owned and then frees the array — so the elements are
315            // moved out and the array released.
316            unsafe {
317                callback::dispatch_once::<F>(userdata, |f| {
318                    // Claimed before the result is inspected: the SDK transfers this
319                    // payload regardless of outcome, so taking it only on success
320                    // would leak it on every failed request.
321                    let guilds = span::take(guilds, |raw| GuildMinimal::from_raw(raw));
322                    f(to_result(result).map(|()| guilds))
323                })
324            }
325        }
326        // SAFETY: the SDK owns the boxed closure and frees it via `free_fn`.
327        unsafe {
328            sys::Discord_Client_GetUserGuilds(
329                self.as_raw_mut(),
330                Some(tramp::<F>),
331                callback::free_fn::<Option<F>>(),
332                callback::once_userdata(callback),
333            )
334        }
335    }

421    /// Invite the current user to the guild owning the lobby's linked channel.
422    ///
423    /// On success the callback receives the invite URL. On desktop the user is
424    /// also forwarded to the Discord client to accept or decline it; on consoles
425    /// no navigation happens, so the game must present the URL itself.
426    ///
427    /// `on_provisional_merge_required` fires if the operation cannot proceed
428    /// until the user's provisional account is merged into a real Discord
429    /// account. It is registered as a persistent handler and stays installed
430    /// until the SDK releases it, so it is [`FnMut`] rather than [`FnOnce`].
431    pub fn join_linked_lobby_guild<M, F>(
432        &mut self,
433        lobby_id: u64,
434        on_provisional_merge_required: M,
435        callback: F,
436    ) where
437        M: FnMut() + 'static,
438        F: FnOnce(Result<String>) + 'static,
439    {
440        unsafe extern "C" fn merge_tramp<M: FnMut() + 'static>(userdata: *mut c_void) {
441            // SAFETY: `userdata` is the boxed `M` installed below.
442            unsafe { callback::dispatch_mut::<M>(userdata, |m| m()) }
443        }
444        unsafe extern "C" fn tramp<F>(
445            result: *mut sys::Discord_ClientResult,
446            invite_url: sys::Discord_String,
447            userdata: *mut c_void,
448        ) where
449            F: FnOnce(Result<String>) + 'static,
450        {
451            // SAFETY: `invite_url` is transferred to us and must be freed, so it is taken.
452            unsafe {
453                callback::dispatch_once::<F>(userdata, |f| {
454                    // Claimed before the result is inspected: the SDK transfers this
455                    // payload regardless of outcome, so taking it only on success
456                    // would leak it on every failed request.
457                    let invite_url = string::take(invite_url);
458                    f(to_result(result).map(|()| invite_url))
459                })
460            }
461        }
462        // SAFETY: the SDK owns both boxed closures and frees each through its
463        // own `free_fn`; the two userdata pointers are never mixed up because
464        // each trampoline is monomorphised over its own closure type.
465        unsafe {
466            sys::Discord_Client_JoinLinkedLobbyGuild(
467                self.as_raw_mut(),
468                lobby_id,
469                Some(merge_tramp::<M>),
470                callback::free_fn::<M>(),
471                callback::persistent_userdata(on_provisional_merge_required),
472                Some(tramp::<F>),
473                callback::free_fn::<Option<F>>(),
474                callback::once_userdata(callback),
475            )
476        }
477    }
