1//! Rich presence, activity invites, join requests, and game launch registration.
2//!
3//! # Rich presence
4//!
5//! Rich presence is what other users see on Discord while this user plays: the
6//! game's name, what they are doing, and how big their party is. Publish it with
7//! [`Client::update_rich_presence`] and remove it with
8//! [`Client::clear_rich_presence`]. The [`Activity`] type carries the fields; see
9//! its documentation for how each one is rendered.
10//!
11//! Rich presence also powers invites, so an activity that is meant to be joinable
12//! must set a party with an id and a size that has room left, and a join secret.
13//!
14//! # Invites and join requests
15//!
16//! Both directions are message-backed and both surface through the *same* pair of
17//! event callbacks, [`on_activity_invite_created`](Client::on_activity_invite_created)
18//! and [`on_activity_invite_updated`](Client::on_activity_invite_updated). The
19//! [`invite_type`](crate::activity::ActivityInvite::invite_type) of the delivered
20//! invite distinguishes them.
21//!
22//! Inviting someone in:
23//!
24//! 1. User A publishes rich presence carrying a join secret.
25//! 2. User A calls [`send_activity_invite`](Client::send_activity_invite).
26//! 3. User B receives it through `on_activity_invite_created`.
27//! 4. User B calls [`accept_activity_invite`](Client::accept_activity_invite) and
28//!    gets the join secret back, which the game uses to place them in the party.
29//!
30//! Asking to be let in:
31//!
32//! 1. User B calls [`send_activity_join_request`](Client::send_activity_join_request).
33//! 2. User A receives an invite of type
34//!    [`ActivityActionType::JoinRequest`](crate::enums::ActivityActionType) through
35//!    `on_activity_invite_created`.
36//! 3. If User A agrees, they call
37//!    [`send_activity_join_request_reply`](Client::send_activity_join_request_reply),
38//!    which sends User B a regular invite.
39//! 4. User B accepts it as above.
40//!
41//! When the user also runs the Discord desktop client, they may instead accept an
42//! invite there. That arrives through
43//! [`on_activity_join`](Client::on_activity_join) or
44//! [`on_activity_join_with_application`](Client::on_activity_join_with_application)
45//! rather than through the invite callbacks.
46//!
47//! # Launching the game from Discord
48//!
49//! Accepting an invite in the Discord client only works if Discord knows how to
50//! start the game. [`register_launch_command`](Client::register_launch_command) and
51//! [`register_launch_steam_application`](Client::register_launch_steam_application)
52//! tell it how. **These write registrations outside this process** — into the OS
53//! registry or equivalent application association store — so their effect outlives
54//! the running program. Call one of them at startup.

63/// Trampoline shared by the callbacks that report only success or failure.
64///
65/// Covers `Discord_Client_SendActivityInviteCallback`,
66/// `Discord_Client_UpdateRichPresenceCallback`, and
67/// `Discord_Client_OpenConnectedGamesSettingsInDiscordCallback`, which are all the
68/// same shape.
69unsafe extern "C" fn result_tramp<F>(result: *mut sys::Discord_ClientResult, userdata: *mut c_void)
70where
71    F: FnOnce(Result<()>) + 'static,
72{
73    // SAFETY: `userdata` is the boxed `F` installed alongside this trampoline.
74    unsafe { callback::dispatch_once::<F>(userdata, |f| f(to_result(result))) }
75}

77/// Trampoline shared by both `Discord_Client_ActivityInviteCallback` sites.
78///
79/// The invite is **transferred** to the callback, not lent: the official C++
80/// wrapper adopts it as `DiscordObjectState::Owned` and lets its destructor run
81/// when the handler returns. It is therefore taken with `from_raw` and dropped
82/// here; borrowing it instead would leak one invite per event.
83///
84/// The handler still receives it by reference so the common case — reading a few
85/// fields — needs no clone. Handlers that must keep the invite can clone it.
86unsafe extern "C" fn invite_tramp<F>(invite: *mut sys::Discord_ActivityInvite, userdata: *mut c_void)
87where
88    F: FnMut(&ActivityInvite) + 'static,
89{
90    // SAFETY: `userdata` is the boxed `F` installed below. `invite` is an owned
91    // handle transferred to us; reading it out by value and wrapping it in
92    // `ActivityInvite` makes the local responsible for dropping it.
93    unsafe {
94        callback::dispatch_mut::<F>(userdata, |f| {
95            let owned = ActivityInvite::from_raw(std::ptr::read(invite));
96            f(&owned)
97        })
98    }
99}

104    /// Publish rich presence for the current user.
105    ///
106    /// Other users on Discord then see that this user is playing, along with
107    /// whatever hints the [`Activity`] carries — a character name, a map, a score.
108    /// Rich presence is also what makes activity invites possible.
109    ///
110    /// The activity is a *partial* object: `name` and `application_id` cannot be
111    /// set here and are overwritten by the SDK.
112    ///
113    /// On desktop this may be called before [`connect`](Client::connect), but
114    /// connecting clears it; while disconnected it sets the presence directly in
115    /// the user's Discord client, when one is available.
116    ///
117    /// Completes through [`run_callbacks`](crate::run_callbacks).
118    pub fn update_rich_presence<F>(&mut self, activity: &mut Activity, callback: F)
119    where
120        F: FnOnce(Result<()>) + 'static,
121    {
122        // SAFETY: `activity` is read during the call and not retained; the SDK owns
123        // the boxed closure and frees it via `free_fn`.
124        unsafe {
125            sys::Discord_Client_UpdateRichPresence(
126                self.as_raw_mut(),
127                activity.as_raw_mut(),
128                Some(result_tramp::<F>),
129                callback::free_fn::<Option<F>>(),
130                callback::once_userdata(callback),
131            )
132        }
133    }

143    /// Send an activity invite to `user_id`.
144    ///
145    /// The invite travels as a Discord message, so it only goes through when at
146    /// least one of the following holds:
147    ///
148    /// - both users are online, in the game, and have not blocked each other;
149    /// - both users are friends;
150    /// - both users share a Discord server and have previously DM'd each other.
151    ///
152    /// `content` is optional message text to accompany the invite; an empty string
153    /// is fine.
154    pub fn send_activity_invite<F>(&mut self, user_id: u64, content: &str, callback: F)
155    where
156        F: FnOnce(Result<()>) + 'static,
157    {
158        // SAFETY: the content string is copied by the SDK during the call.
159        unsafe {
160            sys::Discord_Client_SendActivityInvite(
161                self.as_raw_mut(),
162                user_id,
163                string::borrow(content),
164                Some(result_tramp::<F>),
165                callback::free_fn::<Option<F>>(),
166                callback::once_userdata(callback),
167            )
168        }
169    }

215    /// Accept an activity invite this user received.
216    ///
217    /// The callback receives the join secret from the inviting user's rich presence
218    /// activity, which the game uses to place this user into its own party system —
219    /// for example by passing it as the secret when creating or joining a lobby.
220    ///
221    /// `invite` may be borrowed straight from
222    /// [`on_activity_invite_created`](Self::on_activity_invite_created).
223    pub fn accept_activity_invite<F>(&mut self, invite: &ActivityInvite, callback: F)
224    where
225        F: FnOnce(Result<String>) + 'static,
226    {
227        unsafe extern "C" fn tramp<F>(
228            result: *mut sys::Discord_ClientResult,
229            join_secret: sys::Discord_String,
230            userdata: *mut c_void,
231        ) where
232            F: FnOnce(Result<String>) + 'static,
233        {
234            // SAFETY: `join_secret` is transferred to us and must be freed,
235            // so it is viewed rather than taken.
236            unsafe {
237                callback::dispatch_once::<F>(userdata, |f| {
238                    let outcome = to_result(result).map(|()| string::take(join_secret));
239                    f(outcome)
240                })
241            }
242        }
243        // SAFETY: the C signature takes a non-const `Discord_ActivityInvite*` but
244        // only reads it, matching the const-cast the official C++ wrapper performs
245        // at this same call site. The invite is not retained past the call.
246        unsafe {
247            sys::Discord_Client_AcceptActivityInvite(
248                self.as_raw_mut(),
249                invite.raw_ptr(),
250                Some(tramp::<F>),
251                callback::free_fn::<Option<F>>(),
252                callback::once_userdata(callback),
253            )
254        }
255    }

259    /// Be notified when this user receives an activity invite.
260    ///
261    /// Invites are sent as Discord messages, and the SDK parses them out: the
262    /// message-created callback does *not* fire for them. The delivered invite
263    /// carries everything needed to identify it later for
264    /// [`accept_activity_invite`](Self::accept_activity_invite).

266    /// Join requests arrive here too, distinguished by
267    /// [`ActivityActionType::JoinRequest`](crate::enums::ActivityActionType).
268    ///
269    /// The invite is **borrowed**: it belongs to the SDK and is freed as soon as
270    /// the handler returns. To keep it, clone it out with

273    where
274        F: FnMut(&ActivityInvite) + 'static,
275    {
276        // SAFETY: the SDK owns the boxed closure and frees it via `free_fn`.
277        unsafe {
278            sys::Discord_Client_SetActivityInviteCreatedCallback(
279                self.as_raw_mut(),
280                Some(invite_tramp::<F>),
281                callback::free_fn::<F>(),
282                callback::persistent_userdata(callback),
283            )
284        }
285    }

312    ///

335        unsafe {
336            sys::Discord_Client_SetActivityJoinCallback(
337                self.as_raw_mut(),
338                Some(tramp::<F>),
339                callback::free_fn::<F>(),
340                callback::persistent_userdata(callback),

342        }

382    /// Tell Discord how to launch this game, so invites accepted in the Discord
383    /// client can start it.
384    ///
385    /// Returns whether the command was registered.
386    ///
387    /// **This registration lives outside the process.** It is written to the
388    /// operating system's application registry, so it persists after the game
389    /// exits and affects how Discord launches the game in future sessions. Call it
390    /// when the SDK starts up.
391    ///
392    /// On Windows and Linux `command` is a path to an executable and may include
393    /// launch parameters, for example `"C:\path\to my\game.exe" --full-screen`.
394    /// Passing an empty string registers the currently running executable. To
395    /// launch through a custom protocol such as `my-awesome-game://`, pass that as
396    /// an argument of the executable that handles it.
397    ///
398    /// On macOS the game must be bundled — `command` must be a `.app`, or a custom
399    /// protocol, but *not* a path to a bare executable. An empty string registers
400    /// the currently running bundle, if any.
401    pub fn register_launch_command(&mut self, application_id: u64, command: &str) -> bool {
402        // SAFETY: the command string is copied by the SDK during the call.
403        unsafe {
404            sys::Discord_Client_RegisterLaunchCommand(
405                self.as_raw_mut(),
406                application_id,
407                string::borrow(command),
408            )
409        }
410    }

463    /// Check whether the Discord mobile app is installed on this device.
464    ///
465    /// Always reports `false` on desktop platforms. Requires no connection and may
466    /// be called at any time — useful for deciding whether to offer app-based
467    /// authorization or fall back to a browser flow.
468    ///
469    /// Platform requirements:
470    ///
471    /// - iOS: the app's `Info.plist` must list `"discord"` in
472    ///   `LSApplicationQueriesSchemes`.
473    /// - Android: the manifest must list `"com.discord"` in its `queries` element
474    ///   (required on Android 11 and later).
475    ///
476    /// Despite being a local check this is delivered asynchronously, through
477    /// [`run_callbacks`](crate::run_callbacks).
478    pub fn is_discord_app_installed<F>(&mut self, callback: F)
479    where
480        F: FnOnce(bool) + 'static,
481    {
482        unsafe extern "C" fn tramp<F>(installed: bool, userdata: *mut c_void)
483        where
484            F: FnOnce(bool) + 'static,
485        {
486            // SAFETY: `userdata` is the boxed `F` installed alongside this trampoline.
487            unsafe { callback::dispatch_once::<F>(userdata, |f| f(installed)) }
488        }
489        // SAFETY: the SDK owns the boxed closure and frees it via `free_fn`.
490        unsafe {
491            sys::Discord_Client_IsDiscordAppInstalled(
492                self.as_raw_mut(),
493                Some(tramp::<F>),
494                callback::free_fn::<Option<F>>(),
495                callback::once_userdata(callback),
496            )
497        }
498    }
