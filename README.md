# discord_social_sdk

Rust bindings to the [Discord Social SDK](https://discord.com/developers/docs/discord-social-sdk/overview).

📖 **[API documentation](https://safeshows.github.io/discord_social_sdk/)**

Two crates:

| Crate | What it is |
| --- | --- |
| `discord_social_sdk_sys` | Raw FFI. Generated by `bindgen` from `cdiscord.h`; covers all 513 exported functions. Entirely `unsafe`. |
| `discord_social_sdk` | Safe wrapper. RAII handles, `Option` for optional fields, `Result` for fallible calls, closures for callbacks. |

## Getting the SDK

These crates bind a **prebuilt** SDK that Discord distributes separately; it is not
vendored here. Download it from your application's page in the
[Discord Developer Portal](https://discord.com/developers/applications), then either:

- unpack it to `discord_social_sdk/` at the workspace root (the default), or
- point `DISCORD_SOCIAL_SDK_DIR` at wherever you unpacked it.

The build script locates the SDK, links the right library for the target, and copies
the shared library next to the built binary so `cargo run` and `cargo test` work
without any manual staging.

Supported targets: Windows (`.lib` + `.dll`), Linux (`.so`), and macOS (`.dylib`, or
the `.xcframework` slice when no flat dylib is present). Enable the `krisp` feature
to link Discord's Krisp noise-cancellation build and stage its `.kef` model files.

## Usage

The SDK is callback-driven and single-threaded. Nothing happens until
`run_callbacks()` is pumped, and every callback is delivered on the thread that
calls it:

```rust,no_run
use discord_social_sdk::{Client, enums::ClientStatus, run_callbacks};

let mut client = Client::new();
client.set_application_id(YOUR_APP_ID);

client.on_status_changed(|status, error, detail| {
    if let Some(error) = error {
        eprintln!("connection error: {error:?} ({detail})");
    }
    if status == ClientStatus::Ready {
        println!("connected");
    }
});

client.connect();

loop {
    run_callbacks();
    std::thread::sleep(std::time::Duration::from_millis(16));
}
```

Authorization follows OAuth2 with PKCE — see the `client::auth` module docs for the
full flow.

## How the C API maps to Rust

The C API is regular enough that the mapping is mechanical:

| C | Rust |
| --- | --- |
| `struct T { void* opaque; }` + `_Init`/`_Drop`/`_Clone` | a struct with `Drop` and `Clone` |
| `Discord_String` out-param | `String` (the SDK's buffer is freed for you) |
| `bool Get(self, T* out)` | `Option<T>` |
| `Set(self, T* value)` with null meaning "clear" | `Option<T>` argument |
| `Discord_XSpan` out-param | `Vec<X>` |
| `Discord_ClientResult` | `Result<T, Error>` |
| `(callback, userDataFree, userData)` | a boxed closure, freed by the SDK |

Two rules the wrapper enforces so you do not have to:

- **Strings passed in are borrowed, strings passed out are owned.** The SDK copies
  its inputs, and the wrapper frees every buffer it hands you.
- **Everything a callback receives is owned, not lent** — handles, strings, spans,
  and the `ClientResult` itself. Each must be released exactly once, and the
  wrapper does that for you. (This is easy to get backwards: the natural
  assumption is that a callback argument is borrowed for the duration of the call.
  It is not, and treating it that way leaks on every event.)

### A caveat worth knowing

`Client::add_log_callback` and `add_voice_log_callback` are the one place the SDK
does not promise to release your closure, and the two SDK builds disagree: the
release library frees it when the client is dropped, the debug library keeps it
for the whole process. Write log handlers that are fine either way — keep their
captures small and free of anything lifetime-sensitive.

## Threading

Wrapper types are neither `Send` nor `Sync` by design: the SDK expects to be driven
from a single thread. `set_free_threaded()` opts the underlying SDK into its
thread-safe mode, which only matters if you use the `sys` crate directly.

## Panics across FFI

A panic inside one of your callbacks would otherwise abort the process, since
unwinding out of an `extern "C"` function is not allowed. The wrapper catches
panics at the boundary, reports them on stderr, and keeps the SDK running.

## Documentation

Docs are hosted on [GitHub Pages](https://safeshows.github.io/discord_social_sdk/),
not docs.rs.

docs.rs builds every crate itself, in a sandbox with no network access — and this
crate cannot be built without Discord's SDK, whose header is not ours to
redistribute inside a published package. So there is nothing for docs.rs to
compile against, and it is left unconfigured rather than configured to fail.

To rebuild and publish the docs from a machine that has the SDK:

```bash
./scripts/publish-docs.sh          # build and stage on the gh-pages branch
./scripts/publish-docs.sh --push   # ...and push it
```

The script replaces the branch contents wholesale, adds the redirect and
`.nojekyll` that Pages needs, and is a no-op when nothing changed. Pushing is
opt-in because it rewrites the whole branch.

Enable Pages once, under **Settings → Pages → Deploy from a branch → `gh-pages` / root**.

Locally, `cargo doc --open` works as usual.
